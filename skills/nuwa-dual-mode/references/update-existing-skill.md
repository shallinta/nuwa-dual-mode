# 更新已有 Skill 增量流程

> 参考 nuwa-skill 原版的「更新已有 Skill」机制,适配 dual-mode 架构。
>
> **核心原则**:不重写整个 skill,只跑必要 Agent 拉最新动态,对比新旧 → 补充 / 修正 / 新增 / 标注变化。

---

## 触发条件

满足以下任一即走更新流程,**不再走完整 Phase 0~5**:

- 用户显式说「更新 XX 的 skill」「XX 最近有新动态」「刷新 XX」「XX 又出新作了,更新一下」
- Phase 0A Step 1 扫描 `.claude/skills/` 发现已存在 `[name]-perspective` / `-persona` / `-toolkit` 同名 skill,且用户没明说"我要重做"

---

## Phase U.0:已存在 skill 三选一询问(必做)

检测到同名 skill 时,**必须**先问用户:

```
检测到本地已存在 skill:`[skill-key]`
- 上次创建:[读取 SKILL.md 的「上次更新」元数据]
- 上次调研截止:[YYYY.MM]

请选择处理方式:
① **覆盖重做**(从 Phase 0 完整走一遍,旧版替换)
② **增量更新**(只跑最新动态 Agent,保留原有框架,补充/修正/新增 → 推荐)
③ **新建 v2 并存**(保留旧版,新版后缀 -v2,例 `naval-perspective-v2`)

请回复 ①/②/③。
```

- 选 ① → 重走 Phase 0A,Phase 0.5 覆盖旧目录
- 选 ② → 进入下方 Phase U.1
- 选 ③ → 走完整 Phase 0~5,skill key 自动加 `-v2`(或 v3、v4)

---

## Phase U.1:读取现有 SKILL.md + 时间差标注

1. Read `[skill 目录]/SKILL.md`,提取关键元数据:
   - 创建日期 / 上次更新 / 调研截止日期
   - description 中的全部触发关键词
   - 心智模型 / 招式清单 / 招式名
   - 已注入的特殊 section(关怀红线 / 时代锚定 / 话题边界 / 同人模块 / 分级免责)
2. 计算时间差:「距今 X 个月,需要拉取这段时间的新动态」
3. Read `[skill 目录]/references/research/06-timeline.md`,锁定最后一条事件作为「上次覆盖到的时间锚点」

---

## Phase U.2:精简 Agent 调研(只跑 3 个,不重启 6 个)

只 spawn 3 个增量 Agent:

| Agent | 目标 | 输出文件 |
|-------|------|---------|
| **Agent 2(最新对话)** | 该人物自上次调研以来的新长访谈 / 播客 / 直播切片 | 追加到 `02-conversations.md` 末尾,加日期分隔符 |
| **Agent 5(最新决策)** | 自上次调研以来的新公开决策 / 新作品 / 立场表态 / 商业动作 | 追加到 `05-decisions.md` 末尾 |
| **Agent 6(时间线更新)** | 自上次调研以来的关键里程碑(尤其最近 12 个月) | 追加到 `06-timeline.md` 末尾 |

**Agent prompt 模板**(以 Agent 6 为例):

```
你的任务:为已有 skill `[skill-key]` 增量更新时间线。
- 上次调研截止:[YYYY.MM]
- 当前日期:[YYYY-MM-DD]
- 只搜:[YYYY.MM] 至今的新动态(著作 / 决策 / 公开表态 / 商业动作 / 立场变化 / 失败案例)
- 不要重复已有信息——只追加增量
- 输出:追加到 `06-timeline.md` 末尾,加分隔符 `## 增量更新 [YYYY-MM-DD]`
- 信息源黑/白名单照旧(按 Phase 0.5 已锁定的地域清单)
```

**剩余 Agent(1 著作 / 3 表达 / 4 他者)默认跳过**,除非用户明说"也补一下著作 / 也补一下评论"。

---

## Phase U.3:新旧对比 + 决策

对增量产物执行 4 类判断:

| 新信息 vs 现有 | 处理 |
|---------------|------|
| **强化现有模型**(新案例支撑旧观点) | 补充到对应 mode-toolkit/case-library.md,不改框架 |
| **与现有模型矛盾**(立场变化 / 观点演化) | 在 `references/shared/values-and-antimodes.md` 加「观点演化」section,标注「早期 X → 近期 Y(YYYY 起)」 |
| **出现新的思维模式 / 招式** | 评估是否需新增心智模型(三重验证 + 招式验证4):通过 → 加入 `key-concepts.md`;不通过 → 仅记入 `decision-heuristics.md` |
| **关键事件 / 失败案例** | 加入 `case-library.md`,并在 `诚实边界` 反映新教训 |

---

## Phase U.4:更新 SKILL.md 元数据

无论以上是否触发修改,**SKILL.md 这 4 处必须更新**:

1. `frontmatter description` 末尾的「创建于 YYYY-MM-DD」→ 改为「上次更新 YYYY-MM-DD,创建于 [原日期]」
2. 「⚠️ 时效性声明」section 的「上次更新」字段 → 当天日期
3. 「⚠️ 时效性声明」section 的「最近重大事件覆盖」字段 → 增量产物中最关键的 1-2 条
4. 「⚠️ 时效性声明」section 的「调研截止」字段 → 当天日期(若拉到了新数据)

---

## Phase U.5:验证 + 交付(轻量版)

- 不重跑 Phase 4 全量 6 项验证(框架未变)
- **只跑 1 个 Sanity 验证**:用更新内容里的 1 个新事实问该 skill,确认它会用更新后的资料回答而非旧资料
- 跑 `scripts/quality_check.py` 自动检查
- 用户确认 → 重新打包 → 按当前 runtime 的 skill 安装约定替换该 skill 目录(Claude Code 下即用新内容覆盖 `~/.claude/skills/[name]/`)

**交付话术**:

```
✅ 已更新 skill:`[skill-key]`

📝 本次更新摘要:
   - 新增时间线条目:N 条(最近一条 [YYYY-MM-DD] [事件])
   - 新增 / 调整心智模型:[X 个 / 无]
   - 新增 / 调整招式:[X 个 / 无]
   - 标注的观点演化:[N 处 / 无]
   - 调研截止刷新到:[YYYY-MM-DD]

🎯 试一下:[针对更新内容设计的 1 个示例问法]

⚠️ 提醒:框架结构未变,触发关键词 / 模式判断 / 使用方式与原版相同。
```

---

## 不该走更新流程的场景(应走完整重做)

- 用户说「我对这个 skill 不满意,重做」
- 已存在 skill 上次更新距今 > 24 个月(信息过旧,增量更新无法回填那么多)
- 蒸馏对象本身发生重大转向(如人物转行 / 巨大丑闻 / 大量旧观点被本人推翻)→ 提醒用户「这种情况下增量更新会保留过时框架,建议覆盖重做」

---

## 与其他文件的接口

- Phase 0A Step 1 已存在检测 → 命中后跳到本文件
- Phase 1 Agent 配置 → 本文件只用 Agent 2/5/6,不全启 6 个
- 时效性元数据 → 双模式 / 单模式模板均已包含「上次更新」字段
- 卸载流程 → 主 SKILL.md「Skill 卸载流程」section
