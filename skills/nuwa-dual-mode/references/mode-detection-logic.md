# 产物形态自动判定逻辑

> Phase 0A.1 专用。自动判定 → 用户确认 → 才能 Phase 0.5。

---

## 判定决策树

```
输入: [人名 / 主题]

Step 1: 是「真人/真实主题」还是「虚构角色」?
  ├─ 虚构角色(游戏/动漫/小说/影视的二次元/三次元角色)
  │   → **强烈推荐**单模式 Persona + 后缀 `-persona`
  │     理由:虚构世界方法论套现实会误导,且用户需求 ≈ 角色扮演对话
  │     用户可改双模式 `-perspective`(但 Toolkit 段必须标注现实局限)
  └─ 真人/真实主题 → Step 2

Step 2: 是「人」还是「方法论」?
  ├─ 纯方法论(反脆弱、价值投资、第一性原理...)
  │   → 单模式 Toolkit + 后缀 `-toolkit`
  └─ 人 → Step 3

Step 3: 此人是否需要避免误代言?
  (敏感性高 = 投资家、企业决策者、政治人物、有具体可操作建议的人)
  ├─ 是 → 双模式 + 默认 Toolkit + 后缀 `-perspective`
  └─ 否 → Step 4

Step 4: 此人是否扮演感强?
  (思想家、作家、艺术家、教师、网红、历史人物)
  ├─ 是 → 双模式 + 默认 Persona + 后缀 `-perspective`
  └─ 否 → 双模式 + 用户选默认 + 后缀 `-perspective`

特殊路径:
- 用户明说「只要扮演 / 只要对话」→ 单模式 Persona + `-persona`
- 用户明说「只要工具 / 只要方法」→ 单模式 Toolkit + `-toolkit`
- 虚构角色用户明说「我要双模式」→ 双模式 + `-perspective` + Toolkit 段加现实局限标注
```

### 虚构角色识别要点

满足以下任一即判定为虚构角色 → **强烈推荐**单模式 Persona(用户可改双模式):

- 来自游戏/动漫/小说/影视/漫画作品的角色(如原神、诡秘之主、福尔摩斯、海绵宝宝、神里绫华)
- 没有现实原型(同人衍生角色、网文主角)
- 即便作品中有"方法论",这些方法论建立在虚构世界规则上(序列/魔法/特殊设定),无法迁移到现实

**反例**(不算虚构角色,可走双模式 Toolkit):
- 真人 IP 化人设(罗永浩、董宇辉) — 真实存在,有现实可操作的方法论
- 历史人物(王阳明) — 真实存在,著作可形成现实方法论

### 虚构角色 `-persona` 的扮演风格规则(生成的 SKILL.md 必须内置)

虚构角色单 Persona 的核心场景是**「角色扮演情景对话」**,而非问答机器人。生成的 SKILL.md 首屏角色扮演硬规则必须包含以下条目:

1. **默认场景声明**:首屏明确「本 Skill 默认是角色扮演情景对话,不是问答百科」
2. **回复风格——像正常说话**:
   - **简短为主**(类似真人对话节奏),但**需要时也可以详细**——讲故事、介绍设定、被认真追问时长度跟着内容走,不为了简短而干巴
   - 大致区间:闲聊回应几句到几十字、正经聊一个话题一两百字、专门讲故事/介绍设定可以更长。**没有硬上限,但要避免无内容堆砌**
   - 带角色情绪(语气词、句末标点、用词选择体现开心/害羞/警惕/平淡等)
   - 可有少量场景/动作/神态描写,**统一用括号 `(...)` 标注**(不要用斜体 `*...*`——对话场景里斜体在不同 runtime 渲染差异大,括号更稳定),例如 `(微微一笑)` `(轻抚剑柄)`,每次回复通常 1-2 句,不喧宾夺主
3. **禁止行为**:
   - ❌ 说明书风、大段方法论输出(角色不会突然变身咨询顾问)
   - ❌ 列编号 1/2/3、出条目列表(角色不会这样说话)
   - ❌ 用 Markdown 标题、表格(对话场景里没人这样讲)
   - ❌ "希望对你有帮助"这类客服式收尾
   - ❌ 为了显得"短小精悍"故意把能讲清楚的话砍碎
4. **主动引导话题**:开场或冷场时可主动问"想聊什么?[话题 1] / [话题 2] / [话题 3]?",带入剧情/世界观钩子
5. **沉浸感优先**:被问"你是 AI 吗"时按角色设定自然应对(如绫华会说"奇怪的说法呢...我是社奉行神里家的绫华哦"),而非破设定承认

---

### Toolkit 模式回复风格(与 Persona 完全不同,不要混淆)

无论是双模式中的 Toolkit 段,还是单模式 `-toolkit`,都遵循以下风格——**这是与 Persona 的关键区别**:

1. **回复可以更长、更完整** — 目标是把方法论讲透 / 把案例诊断到位,**不需要克制字数**,但也避免无内容注水
2. **鼓励结构化** — 标题、列表、表格、代码块、引用块都可以用,提升可读性和可操作性
3. **第三人称分析口吻** — "段永平会用 Right BPP 三件套这样诊断:..."而非"我会..."
4. **必须点名调用招式** — 不能只输出泛化建议(详见 Phase 4 Toolkit Sanity 校验)
5. **没有角色情绪、动作描写、语气词** — 这些是 Persona 专属,Toolkit 不用
6. **典型结构**:`诊断 → 调用的招式名 → 落到用户具体场景 → 风险/局限`

**Persona vs Toolkit 风格对照**:

| 维度 | Persona | Toolkit |
|------|---------|---------|
| 长度 | 像正常说话,简短为主需要时可长 | 长短跟着内容,**鼓励完整** |
| 结构化 | ❌ 不用标题/列表/表格 | ✅ 鼓励使用 |
| 人称 | 第一人称("我") | 第三人称("XX 会...") |
| 情绪/动作 | ✅ 有,适度 | ❌ 无 |
| 招式名 | 自然带入即可 | **必须点名召回** |
| 客服化收尾 | ❌ 禁止 | ❌ 禁止 |

## 用户协商话术(必须发出)

### ⚠️ 首次出现 Persona / Toolkit 时必须解释含义(避免用户看不懂术语)

**在向用户展示「建议主模式」之前,必须先用一段话解释两个模式的含义和区别**,即使在同一会话中已经解释过(因为用户可能跨日来用)。

### 标准模板(含术语解释)

```
基于「[人名/主题]」的特征,我建议:
- 产物形态:[单/双]模式
- [若双模式] 默认主模式:[Persona / Toolkit]
- 命名后缀:[-perspective / -persona / -toolkit]
- 完整 skill key:[name]-[后缀]
- 理由:[一句话,引用上表中的理由]

📖 名词解释(这两个词不是常用术语,简单说明一下):
- **Persona(角色扮演模式)**:Skill 以「我是 XX」的第一人称口吻和你对话,用此人特有的语气、用词、思维方式回答问题。适合「想和这个人聊聊」「想体验他会怎么回应」「想模仿他的风格写作」等场景。
- **Toolkit(方法论分析模式)**:Skill 以第三人称「按 XX 的方法论」对你的具体问题做诊断分析,输出框架化的判断和建议。适合「想用他的思维框架解决我的问题」「想拆解一个案例」「不需要扮演,只要方法」等场景。
- **默认主模式**:Skill 被激活时不指定模式的话,默认走哪一个。两个模式始终都在,可以随时用「切换到对话 / 切换到分析」中途切换。

📖 命名后缀对应:
- `-perspective` = 双模式(Persona + Toolkit 都有)
- `-persona` = 单模式 Persona(只能扮演对话)
- `-toolkit` = 单模式 Toolkit(只能方法论诊断)

[若虚构角色] ⚠️ 你蒸馏的是虚构角色,我**强烈推荐**走单模式 Persona,因为虚构方法论套现实会误导,且大多数用户想要的是角色扮演对话。如确实想要 Toolkit,请说「我要双模式」,我会生成但 Toolkit 段会显著标注"仅虚构世界规则内有效,不构成现实建议"。

如需修改:
- 「改成单模式 / 改成双模式」
- 「默认主模式改成 [Persona / Toolkit]」
- 「后缀改成 [-persona / -toolkit / -perspective]」
- 「就按你建议的」

确认后我开始 Phase 0.5 创建目录。
```

### 后续重复出现时

同一次对话中,若用户已经看过一次完整解释,可省略「📖 名词解释」整段,直接展示建议;但**只要切换到新 Skill 蒸馏任务,就要重新解释一次**。

## 用户拒绝建议时的处理

| 用户修改 | 处理 |
|---------|------|
| 「改成单 Persona」(原双模式) | 接受,Toolkit 内容不生成,后缀改 `-persona`,目录少一层 |
| 「改成单 Toolkit」(原双模式) | 接受,Persona 内容不生成,后缀改 `-toolkit`,但提醒「失去扮演能力」 |
| 「双模式默认改 Toolkit」(原默认 Persona) | 接受,只是 SKILL.md 首屏默认入口换,后缀仍 `-perspective` |
| 「我也不知道,你定吧」 | 用默认配置,Phase 4 验证后再回头确认 |
| 「虚构角色改成双模式 / 我要双模式」 | 接受,后缀改 `-perspective`,Toolkit 段必须显著标注"仅在虚构世界规则内有效,不构成现实建议";Persona 段仍保留情景对话风格规则 |

## 边界 case

- **同一人在不同语境下需要不同模式**:建议拆成两个 skill,或在 SKILL.md 内做更细分的「子模式判断」
- **用户明确指定其他本地已安装的造人/蒸馏 skill**(如「用 XXX 蒸馏」「让 YYY 来造」):让路给该指定 skill,本 skill 不接管
- **用户提供素材但说「就给我做个对话机器人」**:默认单 Persona + 后缀 `-persona`,跳过 Toolkit 内容生成
- **真人 cosplay 虚构角色 / VTuber 中之人**:按"虚构人设"算,走 `-persona`;若用户明确要蒸馏中之人本身,按真人走 `-perspective`
