# 检查点协议(Checkpoint Protocol)

> nuwa-dual-mode 有 **5 个**强制暂停检查点(含 Phase 4.5)。每个检查点的「暂停 → 摘要 → 等待 → 恢复」必须严格执行。

---

## 通用协议

每个检查点必须:
1. **暂停**:不主动进入下一 Phase
2. **摘要**:用表格 / 结构化清单展示当前产物
3. **明示行动**:告诉用户「确认即继续」「想改请说...」
4. **等待**:不要超过 1 条消息就跳过
5. **恢复**:用户确认后,简短回应「收到,进入 Phase X」即可,不重复摘要

---

## Phase 0A.1 检查点:产物形态确认

详见 `mode-detection-logic.md` 的话术模板。

**用户可能的响应** → 处理:
- 「确认 / 就按你说的 / OK」 → 进入 Phase 0.5
- 「改成 [单/双/Persona-only/Toolkit-only]」 → 应用修改,再次确认
- 「为什么不是 X?」 → 简短解释判定逻辑,等待最终选择
- 沉默 / 跑题 → 主动追问「形态选择确认了吗?」

## Phase 1.5 检查点:调研质量摘要

### 摘要表格(自动生成)

```
┌──────────────────┬──────────┬──────────────────────────┐
│ Agent            │ 来源数量  │ 关键发现                  │
├──────────────────┼──────────┼──────────────────────────┤
│ 1 著作           │ N 篇     │ 核心论点: ...             │
│ 2 对话           │ N 段     │ 立场变化: ...             │
│ 3 表达           │ N 条     │ 高频词: ...               │
│ 4 他者           │ N 篇     │ 主要批评: ...             │
│ 5 决策           │ N 个     │ 关键决策: ...             │
│ 6 时间线         │ 完整      │ 最新动态: ...             │
├──────────────────┼──────────┼──────────────────────────┤
│ 矛盾点           │ N 处     │ Agent1 vs Agent4: ...    │
│ 信息不足维度      │ ...      │                          │
│ 一手来源占比      │ X%       │ (要求 >50%)              │
└──────────────────┴──────────┴──────────────────────────┘
```

可调用 `scripts/merge_research.py [skill 目录]` 自动生成。

### 提问

```
调研质量摘要如上。
- 一手来源占比 X%(目标 >50%)
- 矛盾点 N 处(将在「内在张力」中保留)
- 信息不足维度:[列出]

确认进入提炼?或需要补充调研某个维度?
```

### 处理
- 「确认」 → Phase 2
- 「补 Agent X」 → 重启对应 Agent,完成后回此检查点
- 「信息不足维度太多,降低期望」 → 调整为「心智模型 2-3 个 + 加大诚实边界」

## Phase 2.5 检查点:提炼摘要 + 模式归位 + 首屏协商

### ⚠️ 首次出现「首屏」概念时必须解释(避免用户看不懂)

在向用户发出「首屏协商」请求**之前**,必须先用一段话解释「首屏」是什么、为什么重要、用户需要决策什么。同一会话中即使前面解释过,只要重新触达「首屏」相关检查点(Phase 2.5 协商、Phase 3.5 确认),都需要至少**简短重述一次**。

#### 标准解释模板(Phase 2.5 必须发出)

```
📖 名词解释:什么是「首屏」?

「首屏」= SKILL.md 主文件的开头部分(通常前 30-50 行),是 Skill 被激活时 AI 第一眼读到的内容。
- 它就像 App 打开后的第一个画面:决定 AI 拿到 Skill 后**立刻知道什么、必须遵守什么、何时去查更深的内容**。
- 首屏太密(信息塞满)→ AI 读不动,关键规则被淹没。
- 首屏太空(细节都下沉)→ AI 第一次回答就漏掉硬规则(如禁用词、不出戏要求)。
- 因此首屏要做取舍:**必须立刻执行的硬规则上首屏,深度内容下沉到 references/ 子文件,按需加载**。

接下来你要确认的是:哪些模块放在 SKILL.md 首屏,哪些下沉到子文件按需读取。
```

### 三段式摘要

**第一段:提炼结果**
```
- 心智模型:N 个
  1. [名称] — [一句话]
  2. [名称] — [一句话]
  ...
- 决策启发式:N 条
- 表达 DNA:[3 个最关键特征]
- 内在张力:N 对
- 诚实边界:N 条
```

**第二段:模式归位预览**(双模式)
```
| 产物 | 归位 |
|------|------|
| [心智模型 1-N] | mode-toolkit/key-concepts.md |
| [决策启发式] | mode-toolkit/decision-heuristics.md |
| [角色规则] | mode-persona/role-play-rules.md |
| [表达 DNA] | shared/expression-dna.md |
| ...
```

**第三段:SKILL.md 首屏内容协商**

参见 workflow-phases.md 中的「首屏推荐清单」。

### 处理
- 「确认」 → Phase 3
- 「心智模型 X 不准 / 缺少 Y」 → 回 Phase 2.1 调整
- 「归位调整:把 X 提到首屏」 → 记录,Phase 3 执行
- 「单模式不需要 mode-X 内容」 → 跳过相应生成

## Phase 3.5 检查点:首屏内容确认(仅双模式)

### ⚠️ 在展示首屏预览之前必须简短重述「首屏」含义

```
📖 提醒:这里的「首屏」= SKILL.md 主文件开头部分(前 30 行内的可视内容)。
现在 Phase 3 已经按 Phase 2.5 协商的方案把 SKILL.md 写完了,
下面贴出首屏 30 行实际产物,你确认这些内容是否合适放在 AI 第一眼看到的位置。
```

Phase 3 写完 SKILL.md 后,**展示首屏 30 行**给用户:

```
SKILL.md 首屏预览(行 1-30):
---
[贴出实际内容]
---

行数总计:N 行(限制 <200,可上探 <400)
description 字符数:N(仅供参考,**不做硬性限制**;只要语义清晰、首屏可读即可)

确认首屏布局?或需要调整?
```

### 处理
- 「确认」 → Phase 4
- 「调换 X 和 Y 的顺序」 → 编辑后再次预览
- 「描述太长」 → 缩短 description 后再次预览
- 「首屏太密」 → 把某模块下沉到 references

---

## Phase 4.5 检查点:修复循环上限(简短协议)

Phase 4.5 修复循环上限为 **2 轮**。第 2 轮跑完仍有 FAIL → **不可自行交付**,必须暂停询问:

```
⚠️ Phase 4.5 已达 2 轮上限,仍有以下 FAIL 项:
- [项 1]:[原因]
- [项 2]:[原因]

请选择:
① 接受当前版本(我会在「诚实边界」中标注以上薄弱维度)
② 继续打磨(说明方向,会再跑一轮)
```

- 用户选 ① → 写入诚实边界,进入 Phase 5
- 用户选 ② → 按用户指示继续,**轮次累计后不再重复询问,按最终结果交付**

**不无限打磨。**

---

## Phase 5.5 检查点:变更摘要确认

Phase 5 双 Agent 精炼后,展示变更:

```
Phase 5 精炼应用了 N 项改动:

【Agent A(结构)建议】
✓ 应用 1: [改动位置] — [改前→改后]
✓ 应用 2: ...
✗ 未应用: [位置] — 理由: 与其他改动冲突

【Agent B(可操作性)建议】
✓ 应用 1: ...
✗ 未应用: ...

最终行数:N 行
最终 description 字符:N

确认进入打包交付?
```

### 处理
- 「确认」 → 交付
- 「再加一条 X」 → 应用后回检查点
- 「撤销改动 N」 → 回滚后回检查点

---

## 用户失联 / 长时无响应

不主动推进。第一次重发时,简短复述「上次卡在 Phase X 检查点,等你确认 Y」即可。
